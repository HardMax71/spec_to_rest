The `spec-to-rest` binary is built from
[`modules/cli`](https://github.com/HardMax71/spec_to_rest/tree/main/modules/cli)
and runs locally via `sbt "cli/run ‚Ä¶"` or as a GraalVM native image. Every
subcommand exits with a numeric code; CI consumers should branch on these.

## Subcommands

| Command | Purpose |
| --- | --- |
| `check` | Parse, type-check, and lint a spec; no codegen. |
| `verify` | Run the Z3 + Alloy verification engine on a spec. |
| `compile` | Emit a project from a spec. Runs `verify` as a gate by default. |
| `diff` | Show which files would change if `compile` ran against an existing output directory. |
| `test` | Invoke the emitted conformance suite against a running service. |
| `inspect` | Print the IR or Dafny prompt material for one operation (debugging). |
| `synth try` / `synth verify` / `synth verify-all` | LLM-driven Dafny body synthesis (Phase 6). |

## Global flags

These flags are accepted by every subcommand:

- `--verbose` / `-v` ‚Äî show stage timings and per-check breakdowns on stderr.
- `--quiet` / `-q` ‚Äî suppress everything except errors.
- `--color` / `--no-color` ‚Äî force ANSI color on or off. Default is auto: color
  is enabled when stderr is a TTY and `NO_COLOR` is not set. `CLICOLOR_FORCE=1`
  forces color even when piped.

## `compile` flags

```text
sbt "cli/run compile [flags] <spec-file>"
```

- `--framework <id>` / `-f` ‚Äî framework (default `fastapi`). Known:
  [fastapi](/targets/python/fastapi/postgres), [chi](/targets/go/chi/postgres),
  [express](/targets/ts/express/postgres).
- `--db <id>` ‚Äî database dialect (default `postgres`): `postgres`, `sqlite`,
  `mysql`. The accepted set depends on the chosen framework.
- `--lang <id>` ‚Äî language (`python`, `go`, `ts`). Always accepted and
  validated against the framework's supported languages; only _required_ when
  the framework supports more than one language (otherwise inferred). Passing
  a language the framework does not support (e.g. `--lang go --framework
  fastapi`) is a hard error, not a silent fallback.

Any unsupported `(language, framework, database)` combination is rejected
with a typed error that lists what the chosen framework actually supports ‚Äî
the resolver never warns-and-falls-back to a default target.
- `--out <dir>` / `-o` ‚Äî output directory (required).
- `--dry-run` ‚Äî run the full pipeline (parse, verify, build, in-memory emit)
  but do not write anything. Prints a manifest of `create` / `update` /
  `unchanged` / `preserve` actions per file. Exits 0 even when the plan is
  empty.
- `--ignore-verify` ‚Äî skip the verification gate. Emits unverified code with a
  warning. Use only for spec exploration.
- `--no-tests` ‚Äî opt out of test emission. The behavioral + stateful +
  structural test suite (`tests/` directory) and the conformance runner are
  emitted **by default** for every supported target; pass `--no-tests` when
  only the service skeleton is needed. Supported for **all nine targets**
  (`python-fastapi-*`, `ts-express-*`, `go-chi-*` √ó postgres / sqlite /
  mysql); the suite is emitted in each target's native language (pytest +
  Hypothesis + Schemathesis / Vitest + fast-check / `go test` + rapid). See
  [Test Generation](/pipelines/test-generation).
- `--strict-strategies` ‚Äî fail compile when default test emission would
  produce incomplete property strategies. No-op when combined with
  `--no-tests`. See [Test Generation](/pipelines/test-generation).
- `--with-synthesis` ‚Äî splice verified Dafny bodies (translated to the target
  language) into the emitted project. Requires `synth verify` to have populated
  the cache. See [LLM Integration](/synth/llm-integration).
- `--allow-skeletons` ‚Äî fall back to unverified skeleton bodies (from
  `synth verify --fallback`) when the verified cache misses. Generated handlers
  halt at runtime with a Dafny `HaltException` when invoked.
- `--synthesis-partial` ‚Äî use verified bodies where cached; for an
  `LLM_SYNTHESIS` op with no verified body, emit a fail-loud stub (skipped by
  testgen) instead of failing the build.
- `--synthesis-model <name>` ‚Äî model to look up in the cache (default
  `claude-sonnet-4-6`).
- `--synthesis-temperature <num>` ‚Äî temperature to look up in the cache
  (default `1.0`).
- `--synthesis-cache-dir <path>` ‚Äî override the cache root.
- `--dafny-bin <path>` ‚Äî path to the Dafny binary (defaults to `$DAFNY_BIN` or
  `PATH`).
- `--dafny-translate-timeout <sec>` ‚Äî wall-clock timeout for `dafny translate`
  (default 60s).

### `--dry-run` example

```text
$ sbt "cli/run compile --framework fastapi --db postgres --ignore-verify \
       --dry-run --out /tmp/svc fixtures/spec/url_shortener.spec"

  create    .gitignore
  create    Dockerfile
  create    Makefile
  create    alembic/env.py
  create    app/main.py
  create    app/routers/url_mappings.py
  ...
‚úî dry-run: 26 files planned for /tmp/svc (create=26 update=0 unchanged=0 preserve=0)
```

## `diff` flags

```text
sbt "cli/run diff --framework <id> --db <id> --out <dir> <spec-file>"
```

- `--framework`/`--db`/`--lang` ‚Äî same target axes as `compile`.
- `--out <dir>` / `-o` ‚Äî directory to compare against (required).
- `--ignore-verify` ‚Äî compare regardless of whether the spec verifies.
- `--no-tests` ‚Äî exclude test files from drift detection. Matches `compile`'s
  default-on behaviour: test files are included by default; pass `--no-tests`
  to skip them.

`diff` exits **0** when the directory is in sync with what `compile` would
produce, **1** when there is drift. The output lists only files that would be
created or updated; unchanged and user-preserved files are summarised at the
bottom.

## `test` flags

```text
sbt "cli/run test --out <generated-dir> [--profile thorough] [--server-url ...]"
```

- `--out <dir>` / `-o` ‚Äî directory containing the previously emitted project
  (required). `test` globs `tests/run_conformance.*` and dispatches to the
  single file it finds; the file's shebang line declares the interpreter
  (`#!/usr/bin/env python3` / `node` / `bash`), so the wrapper carries no
  per-language mapping.
- `--profile <name>` ‚Äî conformance profile: `smoke` (fast), `thorough`
  (default), `exhaustive` (multi-minute). Forwarded to the runner as
  `argv[1]` and as `SPEC_TEST_PROFILE`.
- `--server-url <url>` ‚Äî base URL for the service under test. **Optional**;
  if omitted, each runner picks its own per-target default
  (`http://localhost:8000` for python-fastapi, `http://localhost:8080` for
  ts-express / go-chi). When set, exported as `SPEC_TEST_BASE_URL`. The
  service must be running with `ENABLE_TEST_ADMIN=1` so the runner can hit
  `/__test_admin__/reset` between phases.
- `--runner-bin <path>` ‚Äî override the interpreter read from the runner's
  shebang.

`test` is a thin uniform wrapper around the per-target runner that
`compile` emits ‚Äî same exit-code contract (0 / 1 / 2) and same
`SPEC_TEST_*` env-var contract across all three languages. The runner
itself is part of the [generated test suite](/pipelines/test-generation).
For a project compiled with `--no-tests`, `test` exits 2 with a hint to
re-compile.

## `verify` flags

See [Verification Pipeline](/pipelines/verification) for the full reference.
The most useful flags for CI:

- `--timeout <ms>` ‚Äî per-check Z3/Alloy timeout (default 30000).
- `--parallel <n>` ‚Äî number of concurrent checks (`0` for serial; default is
  the host's available processors).
- `--json` / `--json-out <path>` ‚Äî emit a machine-readable report to stdout or
  a file (suppresses text output). The `--dump-*` flags print the translation
  and exit before the solver runs.
- `--explain` ‚Äî extract Z3 unsat cores and surface contributing spec spans on
  failures.
- `--dump-smt` / `--dump-alloy` ‚Äî write the translated SMT-LIB or Alloy source
  and exit without running the solvers.

## `check` flags

`check` is the cheapest stage: parse ‚Üí IR ‚Üí lint. It does not invoke Z3 or
Alloy. Use it as a fast pre-commit gate.

```text
sbt "cli/run check fixtures/spec/url_shortener.spec"
```

## `inspect` flags

`inspect` dumps internal compiler state for debugging.

- `--format <fmt>` ‚Äî `summary` (default), `json` (full IR + synthesis tally),
  `ir` (raw IR `toString`), `dafny` (Dafny module skeleton), `dafny-prompt`
  (LLM prompt material per LLM_SYNTHESIS operation).
- `--operation <name>` ‚Äî filter `dafny-prompt` to a single operation.

## Exit codes

| Code | Meaning | Where it comes from |
| ---- | ------- | ------------------- |
| `0` | Success. | All commands. |
| `1` | Spec violation: parse error, type error, lint error, verification failure, or `diff` drift. | `check`, `verify`, `compile`, `diff`. |
| `2` | Generation / translator error: dafny translate failure, missing `run_conformance.py`, infra failure during `test`, translator coverage gap. | `compile`, `test`, `verify`. |
| `3` | Backend or test failure: Z3/Alloy backend crash, conformance test failure (`pytest` non-zero). | `verify`, `test`. |
| `4` | Trust limitation: a check completed but used a soundness-limited approximation. | `verify`. |

CI consumers can treat `0` as pass, `1` as user error to surface verbatim,
`2`‚Äì`3` as infrastructure problems requiring a retry or operator
intervention, and `4` as a soft warning.

## Environment variables

| Variable | Effect |
| --- | --- |
| `NO_COLOR` (any non-empty value) | Disable ANSI color in `--color=auto` mode. Honors [no-color.org](https://no-color.org). |
| `CLICOLOR_FORCE` | Force color even when stderr is not a TTY. |
| `DAFNY_BIN` | Default path for the Dafny binary used by `compile --with-synthesis` and `synth verify`. |
| `ANTHROPIC_API_KEY` / `OPENAI_API_KEY` | Required by `synth try` / `synth verify` for LLM calls. |
| `SPEC_TEST_BASE_URL`, `SPEC_TEST_PROFILE` | Set by `test` when launching `run_conformance.py`; can also be exported manually if invoking the runner directly. |
